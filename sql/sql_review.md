## 문제 1. 작품 완주율 (Completion Rate)

### 문제 요약
- `CONTENTS` (작품 정보), `WATCH_LOGS` (시청 로그) 두 테이블.
- 동일 이용자가 한 작품을 시청한 시간을 **모두 합산**했을 때, 누적 시청 시간이 작품 러닝타임의 **95% 이상**이면 그 이용자는 해당 작품을 **완주**한 것으로 간주.
- 작품을 1분 이상 시청한 고유 이용자 수가 1명 이상인 작품에 대해 다음을 구한다:
  - `TOTAL_VIEWERS`: 작품을 1분 이상 시청한 고유 이용자 수
  - `COMPLETED_VIEWERS`: 완주 요건을 충족한 이용자 수
  - `COMPLETION_RATE`: `COMPLETED_VIEWERS / TOTAL_VIEWERS`를 반올림하여 소수점 둘째 자리까지
- 결과는 `CONTENT_ID` 기준 **내림차순** 정렬.

### 테이블 스키마

**CONTENTS** (작품 정보)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `CONTENT_ID` | INT (PK) | 작품 고유 ID |
| `TITLE` | VARCHAR | 작품 제목 |
| `RUNTIME_MIN` | INT | 작품 러닝타임(분) |

**WATCH_LOGS** (시청 로그)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `LOG_ID` | INT (PK) | 로그 고유 ID |
| `CONTENT_ID` | INT (FK) | 시청한 작품 ID |
| `SUB_ID` | INT | 이용자(구독자) ID |
| `MINUTES` | INT | 해당 로그의 시청 시간(분, 1 이상 정수) |

### 풀이 논리

1. **CTE (`WATCH_SUMMARY`) — 사용자별 누적 시청 시간 계산**
   - `WATCH_LOGS`는 로그 1건이 1행. 같은 사람이 같은 작품을 여러 번 보면 행이 여러 개.
   - `GROUP BY CONTENT_ID, SUB_ID`로 (작품, 이용자) 단위로 묶어 `SUM(MINUTES)` → "이 사람이 이 작품을 총 몇 분 봤나".

2. **INNER JOIN — 시청 기록 없는 작품 자동 제외**
   - `WATCH_SUMMARY`에는 시청 로그가 있는 작품만 존재. 시청자 0명인 작품은 INNER JOIN에서 매칭 안 되어 자동 탈락.
   - 따라서 "이용자 수 1명 이상인 작품" 조건은 INNER JOIN + GROUP BY 구조가 자동으로 충족 → 별도 `HAVING COUNT(*) >= 1` 불필요.

3. **조건부 집계 — 완주자 수 세기**
   - `COUNT(CASE WHEN 조건 THEN 1 END)` 패턴.
   - `CASE`는 조건 충족 시 `1`, 미충족 시 (ELSE 없음) `NULL` 반환.
   - `COUNT(expr)`은 NULL을 세지 않으므로 → 조건을 만족하는 행의 개수만 카운트.

4. **GROUP BY — 작품 단위 집계**
   - 최종 결과는 작품 1개당 1행이어야 하므로 `GROUP BY C.CONTENT_ID`.
   - `C.TITLE`은 PK(`CONTENT_ID`)에 종속이라 MySQL에선 GROUP BY에서 생략 가능하지만, 표준/타 DB 호환을 위해 넣어두는 게 안전.

### 함정 / 주의점
- **`MINUTES`는 1 이상 정수** (지문 명시). 로그가 있으면 누적이 무조건 1분 이상 → "1분 이상 시청" 필터(`HAVING SUM >= 1` 등)는 항상 참이라 효과 없음. 빼도 정답.
- **`ROUND` 안의 `* 1.0`**: 정수 나눗셈 방지용 트릭. 단, MySQL의 `/`는 항상 소수를 반환하므로 불필요. (Oracle 등에선 필요)
- **`SELECT`에 일반 컬럼 + 집계 함수 혼용** → 일반 컬럼은 `GROUP BY`에 들어가야 함 (특히 엄격 모드 DB).

## 문제 2. 신규 출시곡 플레이리스트 (Playlist)

### 문제 요약
- `NEW_RELEASES` 테이블 (`SONG_ID` PK, `ARTIST_NAME`은 가수별 고유 식별자).
- 아래 조건에 따라 플레이리스트를 만들고 `SONG_ID`, `ARTIST_NAME` 조회:
  - 플레이리스트는 최대 **10곡**.
  - 재생 수(`PLAYS`)가 많은 노래 우선.
  - 재생 수 같으면 `SONG_ID`가 작은 노래 우선.
  - 같은 가수의 노래는 최대 **2곡**까지만.
- 결과는 `SONG_ID` 기준 **내림차순** 정렬.

### 테이블 스키마

**NEW_RELEASES** (신규 출시곡)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `SONG_ID` | INT (PK) | 노래 고유 ID |
| `ARTIST_NAME` | VARCHAR | 가수명 (가수별 고유 식별자) |
| `PLAYS` | INT | 재생 수 |

### 풀이 논리

1. **1차 CTE (`ARTISTS`) — 가수당 2곡 제한**
   - `PARTITION BY ARTIST_NAME`으로 가수별 그룹을 나누고, 그 안에서 `PLAYS DESC, SONG_ID ASC` 순으로 순위(`ARTIST_RANK`)를 매김.
   - 이후 `WHERE ARTIST_RANK <= 2`로 각 가수의 상위 2곡만 남김.

2. **2차 CTE (`TOP_TWO`) — 전체 10곡 제한**
   - 가수 제한을 통과한 곡들만 대상으로 다시 **전체 기준** 순위(`OVERALL_RANK`)를 매김 (PARTITION 없음).
   - `WHERE OVERALL_RANK <= 10`으로 전체 상위 10곡 컷.

3. **최종 정렬**
   - 출력은 `SONG_ID DESC`. (곡 선정 기준은 `PLAYS`지만 출력 정렬 기준은 `SONG_ID`라 분리 필요)

### 왜 순서가 중요한가 (가수 2곡 → 전체 10곡)
- "가수당 2곡"을 **먼저** 적용해야 함.
- 만약 "전체 10곡"을 먼저 자르면, 한 가수 곡이 3곡 이상 들어간 채 10곡이 채워져 결과가 틀어짐.
  (예: George 곡이 재생수 상위에 3곡 → 가수 제한을 먼저 안 걸면 3곡 다 후보로 남음)
- 가수 제한은 "후보 자격"을 정하는 1차 조건, 전체 10곡은 자격 통과분 중 최종 컷.

### 왜 `OVERALL_RANK`를 새로 매기나
- 가수당 2곡 필터로 일부 곡이 탈락함.
- 탈락 곡을 **제외한** 남은 곡들 중에서 10등을 잘라야 정확.
- 원본 테이블 기준 순위로는 이미 빠진 곡까지 포함되어 컷이 어긋남.
- → 가수 제한 통과 후(`TOP_TWO`) 단계에서 전체 순위를 **다시** 매겨야 함.

### 함정 / 주의점 (실제로 틀렸던 부분)
- **`ROW_NUMBER()`** — 함수명에 S 붙이지 말 것 (`ROW_NUMBERS` ✗).
- **MySQL `--` 주석은 뒤에 공백 필수** — `--같은` ✗, `-- 같은` ✓. (또는 `#` 사용)
- `ROW_NUMBER` vs `RANK`: 동점을 유일한 순번으로 끊어야 "정확히 N개 자르기"가 되므로 `ROW_NUMBER` 사용. (`RANK`는 동점에 같은 번호 → 개수 어긋남)

## 공통 패턴 정리

| 패턴 | 용도 |
|------|------|
| `GROUP BY ... ` + 집계함수 | 여러 행을 그룹 단위로 압축 |
| `COUNT(CASE WHEN 조건 THEN 1 END)` | 조건부 카운팅 (NULL은 미집계 활용) |
| `ROW_NUMBER() OVER (PARTITION BY ...)` | 그룹별 순위 → 그룹당 상위 N개 추출 |
| `ROW_NUMBER() OVER (ORDER BY ...)` | 전체 순위 → 전체 상위 N개 추출 |
| 다단계 필터링 | 좁은 조건(그룹별) 먼저 → 넓은 조건(전체) 나중 |
| 선정 기준 ≠ 출력 정렬 기준 | 마지막에 정렬용 쿼리 한 겹 추가 |
